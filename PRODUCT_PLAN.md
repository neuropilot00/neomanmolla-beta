# 너만 몰라 제품 검수 및 개발 기획서

작성일: 2026-05-01  
상태: 정적 베타에서 실제 테스트/서버 베타로 넘어가기 전 단계

## 1. 제품 방향

`너만 몰라`는 4-6명이 짧게 즐기는 소셜 추리 파티 게임이다. 핵심 재미는 한 명만 다른 질문을 받고, 모두의 답변을 본 뒤 누가 다른 질문을 받았는지 의심하는 데 있다.

앞으로의 방향은 단순한 질문 게임이 아니라, 사용자가 자기가 좋아하는 주제 안에서 놀 수 있는 팬덤형 파티 게임으로 잡는다.

- 친구끼리: 일상, 연애/썸, MT/술자리 질문팩
- 팬덤끼리: K-pop, V01D, 아티스트별 질문 테마
- 스트리머/커뮤니티: 관전 링크, 시청자 예측 투표
- 장기 확장: 캐릭터 꾸미기, 방 테마, 질문팩, 공식 아티스트 테마

## 2. 현재 구현 검수

현재 구현된 것:

- 모바일 정적 웹앱
- 메인 슬라이드 히어로
- 혼자 플레이 모드
- 4-6인 로컬 파티 흐름
- 방 만들기 화면
- 질문팩 선택
- K-pop 질문팩
- 질문 테마 선택 구조
- V01D 공식 질문 테마
- V01D 공지/등록 현황
- 한국어/일본어 로컬라이징
- 플레이어별 언어 선택
- 전신 캐릭터 꾸미기
- 프로필 탭: 꾸미기 / 최애 / 프레임
- 결과 화면에서 공통 질문/가짜 질문 공개
- 초대 링크에 방 설정 전달

현재 아쉬운 점:

- 실제 멀티플레이 서버가 없다.
- 방 상태, 답변, 투표가 한 기기 안에서만 돈다.
- 관전/시청자 투표 문구와 데이터는 일부 준비됐지만 실제 플로우는 미완성이다.
- 질문 테마가 선택되지만, 아직 질문 내용이 테마별로 충분히 달라지지는 않는다.
- 프로필/아이템은 저장되지만 계정이 없어 기기 기반이다.
- 질문팩 관리 화면은 유저용이며, 운영자용 CMS는 없다.
- 아티스트 정보는 정적 데이터라 최신 공지를 자동 반영하지 못한다.

## 3. 제품 구조 원칙

앞으로 하드코딩을 피하기 위해 다음 구조를 유지한다.

```txt
data/
  game-data.js
    settings
    heroSlides
    notices
    idolGroups
    themes
    dressUp
    packs

services/
  roomService
  contentService
  profileService
  voteService
  audienceService
  noticeService
  analyticsService

ui/
  screens
  components
```

현재는 파일이 분리되어 있지 않더라도, 개발 방향은 이 서비스 단위로 간다. UI는 `roomService.createRoom()` 같은 인터페이스만 호출하고, 실제 구현은 LocalStorage에서 Railway 서버 API로 갈아끼울 수 있어야 한다.

## 4. 우선 개발해야 할 기능

### 4.1 실제 방 서버

목표:

- 여러 기기에서 같은 방에 들어와 동시에 플레이
- 방장, 참가자, 관전자 역할 분리
- 답변/투표/결과가 실시간 동기화

필요 API:

```txt
POST /rooms
GET /rooms/:code
POST /rooms/:code/join
POST /rooms/:code/start
POST /rooms/:code/answers
POST /rooms/:code/votes
GET /rooms/:code/state
```

추천:

- Railway 신규 서비스
- Node.js + Express + Socket.IO
- PostgreSQL 또는 Redis-like ephemeral room store

### 4.2 질문팩/질문 테마 시스템

현재 `K-pop 질문팩`과 `V01D 질문 테마`가 있다. 다음 단계는 질문팩과 테마를 분리하는 것이다.

구조:

```txt
질문팩: K-pop 팬덤
질문 테마:
  - 전체 그룹
  - V01D
  - BTS
  - BLACKPINK
  - ...
```

개발 필요:

- 테마별 질문 override
- 테마별 정답 후보/봇 답변 후보
- 언어별 질문 동기화
- 공식 등록 아티스트 우선 노출
- 질문팩 버전 관리

### 4.3 공지/최신정보

목표:

- 내 최애 그룹이 등록됐는지 확인
- 새 질문 테마 추가 공지
- 신규 아티스트/컴백/이벤트 질문팩 안내

필요 API:

```txt
GET /notices
GET /artists
GET /artists/:id
```

운영 방향:

- `V01D 등록 완료`처럼 공식 등록 아티스트는 상단 노출
- 등록되지 않은 그룹은 “요청하기” 버튼 제공
- 나중에 관리자 페이지에서 공지 추가

### 4.4 프로필/꾸미기

현재 전신 캐릭터는 좋은 출발점이다. 다만 진짜 판매/재방문 요소가 되려면 아래가 필요하다.

개발 필요:

- 파츠 슬롯 확장: 헤어, 상의, 하의, 악세서리, 손 아이템, 오라, 배경
- 세트 코디: V01D 테마 세트, 무대 세트, 팬덤 세트
- 저장 슬롯
- 방 입장 시 캐릭터 표시
- 결과 카드에 캐릭터 반영

중요:

- 프로필의 최애 그룹은 개인화 용도다.
- 게임의 질문 테마 선택은 방 만들기 쪽이 메인이다.

### 4.5 관전/시청자 투표

이 기능은 스트리머/팬덤 확장에 중요하다.

목표:

- 관전자 링크 생성
- 관전자는 답변을 보고 가짜를 예측
- 결과 화면에서 플레이어 투표와 시청자 투표를 함께 표시

필요 API:

```txt
POST /rooms/:code/audience/votes
GET /rooms/:code/audience/result
```

UX:

- 관전자는 질문 원문을 바로 보지 않음
- 답변 공개 후 예측 투표
- 결과 때 “시청자 68%가 V01D 팬을 의심” 같은 카드 표시

### 4.6 결과 카드/공유

현재 결과 카드가 텍스트 중심이다. 팬덤 게임은 공유 카드가 중요하다.

개발 필요:

- 이미지 결과 카드 생성
- 방 테마/캐릭터/질문 테마 반영
- “오늘의 가짜”, “시청자 예측률”, “질문 차이” 포함
- 카카오/라인/트위터 공유 고려

### 4.7 운영자용 콘텐츠 관리

정식 테스트를 하려면 질문을 코드에 직접 추가하면 안 된다.

필요:

- 질문팩 관리
- 아티스트 등록 관리
- 공지 관리
- 금칙어/신고 관리
- 언어별 번역 관리

초기에는 Google Sheet 또는 Notion을 관리 원본으로 두고, 서버가 JSON으로 내려주는 방식도 가능하다.

관리자 페이지 필요 여부:

- 내부 소규모 베타: 없어도 된다. 대신 `admin.json`, Google Sheet, Notion DB 같은 운영 원본이 필요하다.
- 외부 베타: 최소 관리자 페이지가 필요하다. 공지 추가, 질문팩 on/off, 아티스트 등록 상태 변경 정도는 개발자 없이 바꿀 수 있어야 한다.
- 정식 공개: 관리자 페이지가 필수다. 권한, 승인, 버전 관리, 신고 대응, 번역 상태 확인이 없으면 운영이 막힌다.

최소 관리자 페이지 기능:

```txt
/admin
  로그인
  공지 작성/수정/숨김
  아티스트 등록/수정/공개 여부
  질문팩 공개/비공개
  질문 테마 추가/수정
  신고 목록 확인
```

초기에는 `admin`에서 바로 DB를 수정하지 않고, `draft -> review -> publish` 상태를 두는 것이 좋다. 질문 하나가 이상하면 게임 전체 품질이 깨지기 때문이다.

## 5. 화면 구조 제안

사용자 피로를 줄이기 위해 한 화면에 한 목적만 둔다.

메인:

- 히어로 슬라이드
- 혼자 플레이
- 방 만들기
- 내 프로필
- 질문팩
- 공지

방 만들기:

- 인원
- 질문팩
- 질문 테마
- 방 테마
- 초대 링크

프로필:

- 꾸미기
- 최애
- 프레임

질문팩:

- 질문팩 목록
- 선택된 질문팩의 질문 테마
- 공식 등록 아티스트 우선 노출

공지:

- 최신 등록 아티스트
- 신규 질문팩
- 이벤트/업데이트

설정:

- 언어
- 기본 방 테마

### P0: 실제 테스트 가능

2. 방 생성/참가 API
3. Socket.IO 실시간 상태 동기화
4. 답변/투표/결과 동기화
5. 기존 정적 앱을 API 기반으로 연결
6. 운영용 `admin.json` 또는 Google Sheet 연동

### P1: 팬덤 게임화

1. 질문 테마별 질문 데이터 분리
2. V01D 전용 질문 20개 이상 추가
3. 공식 아티스트 등록 화면 개선
4. 공지 API 연결
5. 결과 카드 개선
6. 최소 관리자 페이지 추가

### P2: 재방문/수익 기반

1. 계정 또는 닉네임 기반 프로필 저장

5. 시청자 투표

### P3: 운영/확장

## 9. 결론

다음 개발의 핵심은 UI를 더 화려하게 만드는 것이 아니라, 정적 데모를 실제 방 기반 서비스로 바꾸는 것이다.

가장 먼저 해야 할 일:

1. Railway에 `neomanmolla-api` 서버를 새로 만든다.
2. 방/답변/투표 상태를 서버로 옮긴다.
3. 질문팩과 질문 테마를 API에서 내려받게 한다.
4. V01D 같은 공식 아티스트 테마를 콘텐츠 상품 단위로 관리한다.
5. 관전/시청자 투표를 붙여 스트리머와 팬덤 플레이로 확장한다.

이 순서로 가면 단순한 파티 게임에서 팬덤형 소셜 추리 플랫폼으로 확장할 수 있다.
